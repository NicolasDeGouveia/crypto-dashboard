# Data Model: Full Crypto Dashboard with Authentication & Favourites

**Branch**: `002-crypto-dashboard-auth` | **Date**: 2026-03-10

---

## Persisted Entities (Database — Neon PostgreSQL)

### User

Represents a registered account. Managed by Auth.js v5 (the auth adapter owns this
table's schema; do not add application columns directly — extend via a separate
`profiles` table if needed in the future).

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `text` | PRIMARY KEY | UUID generated by Auth.js |
| `email` | `text` | UNIQUE, NOT NULL | User's email address |
| `emailVerified` | `timestamp` | NULLABLE | Set by Auth.js if email verification is added |
| `password` | `text` | NULLABLE | Argon2id hash; null for OAuth-only accounts |
| `name` | `text` | NULLABLE | Display name (optional) |
| `image` | `text` | NULLABLE | Avatar URL (reserved for future OAuth use) |
| `createdAt` | `timestamp` | NOT NULL, DEFAULT now() | Account creation timestamp |

**Notes**:
- The `password` column is an application-level addition to the Auth.js user schema.
  Auth.js itself does not manage passwords — the credentials `authorize` callback
  reads this column directly using Drizzle and compares via argon2.
- Auth.js also creates `accounts`, `sessions`, and `verificationTokens` tables
  (required by the adapter even in JWT mode for OAuth future-proofing). These are
  Auth.js-internal and should not be accessed by application code directly.

### Favourite

Represents a user's saved coin. Simple join table: one user → many coin IDs.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `serial` | PRIMARY KEY | Auto-incrementing integer |
| `userId` | `text` | NOT NULL, FK → users.id ON DELETE CASCADE | Owner |
| `coinId` | `text` | NOT NULL | CoinGecko coin identifier (e.g., `"bitcoin"`) |
| `addedAt` | `timestamp` | NOT NULL, DEFAULT now() | When the coin was favourited |

**Unique constraint**: `(userId, coinId)` — prevents duplicate favourites.

**Index**: `userId` — all queries on this table filter by user; index is essential.

**Notes**:
- `coinId` stores the CoinGecko string ID (e.g., `"bitcoin"`, `"ethereum"`), not a
  numeric ID. This is stable across API calls and does not require a local coins table.
- Deleting a user cascades to delete all their favourites.
- No foreign key to a local `coins` table — coins are always fetched live from the
  CoinGecko API. If a coin ID is retired from the API, the favourite row persists but
  renders a "data unavailable" state.

These are fetched at request time and cached by Next.js ISR. They are never stored in
the database.

## Entity Relationships

```
User (1) ──────< Favourite (N)
                 coinId ──── [CoinGecko API: no local coins table]
```

A `User` has zero or many `Favourite` records.
Each `Favourite` references a coin by its CoinGecko string ID.
Coins are not persisted locally — they are always fetched live.

---

## State Transitions

### Favourite toggle

```
[not favourited] ──── add (POST /api/favourites) ──── [favourited]
[favourited]     ──── remove (DELETE /api/favourites/{coinId}) ──── [not favourited]
```

Both transitions are idempotent (adding an existing favourite or removing a
non-existent one should return success without error).

### Authentication flow

```
[guest] ──── register ──── [authenticated, new user]
[guest] ──── login ──────── [authenticated, existing user]
[authenticated] ──── logout ──── [guest]
[authenticated] ──── session expires (24h) ──── [guest, redirected to /login]
```
